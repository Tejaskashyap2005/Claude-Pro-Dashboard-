📚 Setup guide — making this work for your whole team

1
The auto-reset timer (works out of the box)

When anyone marks an account as "Exhausted," a countdown timer starts automatically. After the set period (default 5 hours, matching Claude's actual token refresh), the account flips back to "Available" on its own. No manual reset needed. You can change the timer duration using the dropdown above.

2
The database problem (so everyone sees the same data)

Right now this file uses localStorage — which means each person's browser has its own copy of the data. If Person A marks an account exhausted, Person B won't see it. To fix this, you need a shared database. The easiest free option is Firebase Realtime Database. Here's how to set it up:

3
Set up Firebase (free, 5 minutes)

a) Go to console.firebase.google.com and create a new project (any name, e.g. "claude-dashboard").

b) Click "Realtime Database" in the left sidebar, then "Create Database." Pick any region. Start in test mode for now.

c) Click the gear icon ⚙ → "Project settings" → scroll to "Your apps" → click the web icon </> → register an app (any nickname). Copy the config object it shows you.

d) Open this HTML file in a text editor. Find the FIREBASE_CONFIG block near the top and replace the placeholder values with your real config:

const FIREBASE_CONFIG = { apiKey: "AIzaSyB1234567890abcdefg", authDomain: "claude-dashboard-abc.firebaseapp.com", databaseURL: "https://claude-pro-dashboard-default-rtdb.firebaseio.com/", projectId: "claude-dashboard-abc", storageBucket: "claude-dashboard-abc.appspot.com", messagingSenderId: "123456789012", appId: "1:123456789012:web:abcdef1234567890" }; // Change this to true: const USE_FIREBASE = true;

e) Save the file. Host it anywhere your team can access — a shared drive, internal server, GitHub Pages, or even email it. Every browser that opens it now reads/writes the same Firebase database in real time.

4
Other database options

Supabase (free tier) — similar to Firebase but uses PostgreSQL. More setup but more powerful queries.

Google Sheets as a backend — use the Sheets API to read/write. Simple but slower and has rate limits.

Your own server — if your company has internal hosting, a simple Node.js/Express app with a JSON file or SQLite database works perfectly.

Notion API — store accounts in a Notion database, query via API. Good if your team already uses Notion.

5
Security note

For production use, update Firebase rules to restrict access. In the Firebase console, go to Realtime Database → Rules and set rules that match your security needs (e.g. require authentication, restrict to your company domain). The test mode rules expire after 30 days.
